PART 5 — Three Action Buttons

Cancel Purchase — always visible and clickable
Save — always visible and clickable
Confirm Purchase — visible but disabled and grayed out until the admin has clicked Verify and the Purchase Order preview has been successfully generated. Only after verification is it enabled and clickable.
Cancel Purchase — Safety Flow
When the admin clicks "Cancel Purchase", do NOT cancel immediately. Instead:
Step 1: Show a popup/modal with:
Warning icon ⚠️
Title: "Are you sure you want to cancel this purchase?"
Message: "This action cannot be undone. Before cancelling, consider whether you can simply edit or modify this purchase instead."
Two buttons: "Go Back & Edit" and "Yes, I want to cancel"
Step 2: If they click "Yes, I want to cancel", show a second confirmation screen with:
Message: "To confirm cancellation, type the code below exactly as shown:"

A text input field where the admin must type that exact code
A "Confirm Cancellation" button — disabled until the typed text matches the generated code exactly (case-sensitive)
A "Go Back" button to abort
Step 3: If the code matches and they confirm, cancel the purchase, clear the form, and show a brief success toast: "Purchase cancelled."
Save — Draft Behavior

Save the current state of the form as a draft purchase in the database with status draft

The admin can then continue filling it, verify, and confirm

The Verify → Confirm Purchase flow still applies the same way

Category flagged as requires_serial = true → show the per-unit row table (IMEI/serial flow)
Category flagged as requires_serial = false → show the simple quantity form

Phones → requires_serial = true
